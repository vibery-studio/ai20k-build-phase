Demo 1 · Query → Chart.

Archetype cho pattern NL query → structured result: text-to-SQL dashboard, voice-to-dashboard, digital twin / what-if simulation, search engine trên knowledge graph, feedback corpus analysis. Cùng một shell: người dùng gõ câu hỏi tiếng Việt → LLM dịch sang SQL (hoặc structured query) → app chạy → kết quả render thành bar chart + table. Khác nhau ở data domain, giống nhau ở cách UI handle Transparency, Control và Recovery.

TL;DR

Ba ý phải nhớ

  • Archetype = pattern "NL prompt in, structured data out". Không phải chat, không phải form. Input tự do, output có schema {sql, rows}.
  • Topic dùng được: text-to-SQL (sales data → chart), digital twin (what-if simulation → chart), feedback analysis (corpus → segmented chart), voice-to-dashboard, search/knowledge graph query. Domain khác, UI giống 80%.
  • T-C-R punchline: Transparency = show SQL (<details>). Control = edit SQL + rerun. Recovery = try/catch + 15s timeout + retry button. Ba thứ này biến AI tool từ "đen kịt" thành "dùng được thật".
Vì sao archetype này

Query → structured result: nơi LLM vừa đủ giỏi, vừa đủ sai.

Đây là nhóm app mà LLM thực sự tạo ra giá trị — và cũng là nhóm mà LLM sai đủ để user không tin nếu UI giấu process. T-C-R ở đây không phải lý thuyết; là 3 lớp bảo hiểm bắt buộc.

Query-to-chart thành công ở nơi người dùng biết mơ hồ mình cần gì nhưng không muốn học SQL. "Doanh thu theo miền trong Q2", "top 5 sản phẩm", "tháng nào churn cao nhất" — câu tiếng Việt tự nhiên. Không ai muốn mở BI tool, drag-drop, filter 4 lần rồi export. Họ muốn gõ câu hỏi, thấy biểu đồ.

Nhưng LLM không đáng tin 100% với SQL. Trong production, text-to-SQL sai khoảng 15-25% lần — join sai bảng, filter sai column, aggregate sai granularity. Sai không phải lỗi rõ ràng kiểu "syntax error"; sai kiểu "SQL chạy ra số, nhưng số đó không phải cái user hỏi". Nếu UI chỉ hiện chart, user tin số sai, ra quyết định sai. Trust gãy một lần, app chết.

Đó là lý do T-C-R tồn tại ở archetype này, không phải "nice-to-have":

  • Transparency: show cái SQL đã gen. User biết gì đã chạy — họ có thể đọc, debug, hoặc ít nhất cảm giác được AI không phải hộp đen.
  • Control: cho user sửa SQL. AI chỉ là first draft. User biết data schema thật của mình hơn LLM — phải cho họ override.
  • Recovery: API fail, timeout, JSON malformed. Tất cả sẽ xảy ra. Retry path phải có.

Ba domain ví dụ (pattern y hệt, khác context):

Text-to-SQL

Dashboard tự phục vụ

User gõ "doanh thu theo sản phẩm Q3", LLM gen SQL trên schema có sẵn, chạy query, render chart. Archetype lõi — 1-1 với demo này.

Digital Twin / what-if

Simulation & decision support

User hỏi "nếu giảm giá 10% thì doanh thu tuần sau?" — LLM gen SQL trên historical data + simulation logic. Output vẫn là chart so sánh baseline vs scenario.

Feedback / corpus analysis

Insight từ dữ liệu định tính

User hỏi "khách phàn nàn nhiều nhất về gì trong tháng 4?" — LLM classify feedback thành category, aggregate, render chart bar theo topic.

Điểm chung ba domain. Input là câu tự nhiên Vietnamese. Output là {sql, rows} shape. Chart = bar chart đơn giản (recharts). Khác biệt duy nhất: schema và system prompt. Đó là lý do một starter adapt được cho cả ba.

0

Setup baseline — header, textarea, chips, chart, table

What it is · scaffold trước khi nói chuyện T-C-R

Vì sao cần riêng: T-C-R chỉ có nghĩa khi có gì đó để thêm vào. Prompt 0 dựng phiên bản minimal — input, button, chart, table — đủ để một VinUni student tin "tôi build được trong 1 ngày". Không SQL visible, không error UI, không edit — đó là chủ ý. Ba thứ sau là teaching moment.

Prompt 0 copy-paste vào Claude Code
# Prompt đầu — baseline (~3-4 phút Claude Code chạy)

Read GUIDELINE.md in this directory end-to-end before writing
anything. Then build the baseline exactly as specified in
section 5 — header, textarea with the Vietnamese placeholder,
Chạy button, three sample chips, loading text, result card
with a recharts BarChart and HTML table. Use the file structure

Đi sâu hơn 1. "Read GUIDELINE.md" là phần quan trọng nhất của prompt. Không có nó, Claude Code sẽ đoán theo training data — và bạn nhận lại một text-to-SQL demo dùng Streamlit, Tailwind, hoặc TypeScript. GUIDELINE là single source of truth. Prompt chỉ nên nói "làm theo spec", không viết lại spec.
Đi sâu hơn 2. Fake fallback 1/8 fail là design deliberate — cho phép Tony demo Recovery mà không cần chờ Gemini thật lỗi. Production code không có cái này; nhưng demo code cần, để rehearsable.
T

Transparency — show cái SQL đã gen

What it is · một <details> tag ở trên chart, collapsed mặc định, expand lần đầu

Vì sao T quan trọng cho archetype này: Text-to-SQL sai ~20% lần. Nếu user không thấy SQL, họ không biết AI đã hỏi cái gì — họ chỉ thấy số ra sai. Mất trust ngay. Ngược lại, khi SQL visible, user đọc được lý do ra sai ("à, nó join sai bảng") và đây thành dữ kiện tin được: AI không phải nói dối, chỉ là draft.

Prompt T transparency
# Prompt T — thêm ~15 dòng code, Claude Code ~30s

Following GUIDELINE.md section 6.T, add a collapsible
<details> block above the chart labeled "SQL đã tạo".
Inside, show the sql string in a <pre><code> block with
monospace font and a light gray background. Collapsed by
default, but open automatically the first time a result
appears so students see the SQL immediately. Don't change
anything else.
Đi sâu hơn 1. Tại sao <details> không phải modal? Modal block UI, force user read SQL trước khi làm gì. <details> = optional cognition — user muốn đọc thì click, không thì không phân tâm. Default collapsed sau lần đầu = user quen rồi không cần reopen mỗi lần.
Đi sâu hơn 2. "Open automatically first time" là nuance quan trọng. Nếu collapsed 100%, nhiều user không bao giờ phát hiện SQL có thể xem. Open lần đầu = một lần awareness trigger. Sau đó họ tự quản lý. Đây là default design có teach, không phải lazy default.

C

Control — cho user sửa SQL rồi rerun

What it is · thay <pre> bằng <textarea> editable + secondary button

Vì sao C quan trọng cho archetype này: AI gen SQL đúng ~80%. 20% còn lại user biết fix nhanh hơn là prompt lại (đổi LIMIT 5 thành LIMIT 10, thêm WHERE region = 'Bắc'). Nếu không có Control, user phải quay lại textarea, rephrase câu hỏi, hy vọng AI hiểu đúng. Friction đủ để user bỏ app. Control = AI là first draft, user là editor-in-chief.

Prompt C control
# Prompt C — thêm ~20 dòng, Claude Code ~45s

Following GUIDELINE.md section 6.C, replace the <pre> SQL view
with an editable <textarea> pre-filled with the SQL. Add a
secondary button labeled "Chạy lại với SQL đã sửa" below
the textarea. When clicked, bypass askLLM and reuse the
existing rows — add a one-line comment noting this is a fake
execution because we have no real DB. The primary Chạy button
still calls askLLM as before.
Đi sâu hơn 1. "Bypass askLLM" và "reuse existing rows" là fake cho demo day. Production version sẽ gọi thẳng backend (FastAPI/Node) để chạy SQL đã sửa trên DB thật. Kiến trúc UI không thay đổi khi chuyển — chỉ đổi thân function. Đó là power của abstraction askLLM.
Đi sâu hơn 2. Tại sao secondary button có style khác primary? Visual hierarchy: "Chạy" (AI) = black, dominant. "Chạy lại với SQL đã sửa" (manual) = outlined, subordinate. User hiểu ngay: flow chính là hỏi-AI; override là escape hatch. Nếu hai button cùng style, user sẽ đắn đo mỗi lần.
R

Recovery — try/catch + timeout + retry button

What it is · red card khi error, retry giữ lại prompt cũ, 15s timeout hard ceiling

Vì sao R quan trọng cho archetype này: LLM APIs fail. Rate limit, 500, malformed JSON, timeout khi hỏi complex. Nếu UI silent fail hoặc hiện "something went wrong", user gõ lại câu hỏi từ đầu — rage quit sau 2 lần. Recovery = error message cụ thể + retry-without-retyping + hard timeout. Ba thứ này quyết định app có khả năng đứng qua ngày production đầu tiên hay không.

Prompt R recovery
# Prompt R — thêm ~25 dòng, Claude Code ~45s

Following GUIDELINE.md section 6.R, wrap the askLLM call in
try/catch. On error, render a red-bordered card containing
the heading "Có lỗi xảy ra", the error message, and a
"Thử lại" button that re-runs the last prompt (keep
the last prompt in state). Also ensure src/llmService.js has
a 15s timeout via Promise.race — add it if missing. Do not
touch the existing Transparency or Control UI.
Đi sâu hơn 1. "Keep the last prompt in state" là nuance easy to miss. Nếu retry button re-render chỉ khi state còn — state phải không bị wipe khi error. Nhiều người implement thành "on error, clear state" — sai. State của câu hỏi phải survive error. Chỉ rows mới clear. Câu hỏi giữ lại.
Đi sâu hơn 2. 15 giây là hard ceiling, không phải guideline. Gemini 2.5 Flash 99% response trong 3s. Nếu > 15s = có gì đó bất thường (network, rate limit, model degraded). Better to fail fast + retry than block UI 30s. Nguyên tắc: "user bỏ cuộc sau 10 giây." 15s là slack cho edge case, không phải target.
Đi sâu hơn 3. Error message phải cụ thể, không "Có lỗi xảy ra. Vui lòng thử lại." generic. Fake mode throw "Fake LLM: simulated failure". Real mode throw "Gemini API timeout (>15s)" hoặc "Invalid response shape". User cần biết đây là lỗi tạm thời hay lỗi schema — quyết định có retry hay không.

— nguyên tắc thiết kế Query → Chart
Stack decisions

React + Vite + plain JSX. Có lý do, không phải mặc định.

Workshop này stack cố ý minimal. Không Next.js, không Tailwind, không TypeScript. Mỗi quyết định có trade-off — dưới đây là reason cho từng pick, và alternative nếu team bạn stack khác.

Vì sao React + Vite

Vite = dev server chạy trong <1s, HMR instant, không cần config. React = student quen từ course đầu. JSX plain (không TS) = đọc từ trên xuống dưới như Python, không cần import type. Hai công cụ này đều zero-config cho use case này.

Vì sao src/llmService.js abstraction

UI không bao giờ import @google/generative-ai trực tiếp. Chỉ import askLLM từ ./llmService. Lý do:

  • Swap provider không sửa UI. Chuyển Gemini → OpenAI → Anthropic = sửa một file. UI không biết.
  • Fake mode inline. Missing API key → fake fallback activate. Student clone repo không có key vẫn chạy được.
  • Test dễ. Mock askLLM trong test thay vì mock fetch + SDK.

Pattern này gọi là "LLM hexagonal" — domain logic không phụ thuộc provider. Khi team bạn chuyển từ dev API key sang production key, hoặc khi Anthropic ra model mới rẻ hơn, chỉ cần edit một file. UI keep running.

Vì sao Gemini 2.5 Flash làm default

  • Rẻ. $0.075 / 1M input tokens — khoảng 1/10 giá OpenAI GPT-4o. Student demo 100 query = ~$0.05.
  • Nhanh. 99th percentile ~3s cho query đơn giản. Đủ nhanh để feel realtime.
  • JSON mode native. responseMimeType: "application/json" — gen SQL + rows structured, không phải regex parse.
  • Vietnamese tốt. Hiểu câu "top 5 sản phẩm bán chạy" không cần rephrase sang English.

So sánh stack thay thế

Lựa chọn Khi nào hợp Trade-off
React + Vite Default cho demo này. Client-only, fast HMR, deploy Vercel/Netlify/GitHub Pages đều chạy. Không có routing built-in — nếu app cần multi-page thì phải add react-router. Cho single-page query UI thì không cần.
Streamlit Team toàn backend Python, không muốn học React. 50 dòng Python = demo chạy. UX kém hơn — không real-time streaming, button hơi clunky, không custom style được nhiều. Demo day show vẫn được nhưng "feel" giống internal tool.
Next.js App lớn hơn, cần routing, SSR, API routes in-repo. Hoặc đã có sẵn Next.js codebase. Overkill cho demo 90 phút. Build time chậm hơn Vite. Setup phức tạp hơn. Nhưng nếu team đã dùng thì copy component sang app/ là chạy.
Gradio Team ML muốn nhanh — demo model ngay, không care UX. UI rất limited. T-C-R implement được nhưng look cheap. Phù hợp cho internal testing, không phải demo day.
OpenAI GPT-4o Team đã có credit OpenAI, hoặc cần reasoning sâu hơn. Đắt hơn Gemini ~10x. Response time slower ~2x. JSON mode có nhưng cần structured outputs schema — thêm config.
Anthropic Claude Sonnet Cần SQL quality cao nhất (complex join, window function). Mid-price. Không có native JSON mode — phải prompt hard cho output shape. Vietnamese tốt nhưng không hơn Gemini đáng kể.

Nguyên tắc chọn stack cho workshop. Stack tốt nhất = stack cho phép ship trong deadline. A20 có 6 tuần, không có thời gian học Next.js + Tailwind + TypeScript mới. React + Vite là điểm ngọt: modern enough để không ngượng demo day, simple enough để không burn time.

Adapt cho team bạn

Cùng shell, ba domain. Thay đúng 5 chỗ là ship.

Starter này adapt cho bất kỳ topic nào thuộc pattern NL → structured. Ba variant tiêu biểu bên dưới: text-to-SQL (lõi), digital twin / what-if (simulation), feedback / corpus analysis. Với mỗi variant, cái gì giữ nguyên (80%) và cái gì đổi (20%). Không đụng đến UI framework, T-C-R structure, hay llmService abstraction — chỉ đổi data shape, system prompt, và sample queries.

Production upgrade path

W4 MVP → W5 backend → W6 polish. Không skip.

Starter là W4. W5 và W6 là upgrade có thứ tự — đừng làm ngược. Mỗi phase có scope rõ, không "overlap cho chắc".

W4 Tuần này · MVP

Clone, adapt, ship local

Fork starter. Đổi system prompt + sample data cho domain. Chạy local với Gemini API key. Deploy Vercel/Netlify. T-C-R hoạt động với fake DB. Demo được cho coach trên Discord trước 27/04.

W5 Tuần sau · Backend thật

Kết nối DB + API

Thay fake execution trong "Chạy lại với SQL đã sửa" bằng call API thật (FastAPI/Node backend của team). askLLM vẫn gen SQL — thay vì rows từ fake, backend chạy query thật trên SQLite/Postgres và trả về. Persist query history nếu cần.

W6 Demo day · Polish

Feedback-driven polish

Mentor + peer review tuần 5 sẽ ra feedback. Ưu tiên fix theo rubric: Business Impact (25%), UX (25%), Agent Functionality (30%). Thêm: input validation, chart title, download CSV button, share link. Không thêm feature mới — chỉ polish existing.

Nếu W5 backend chưa xong → giữ fake. Demo day show UX > show broken backend. Fake data nhìn real nếu Vietnamese names đúng và shape đúng. Rubric không trừ điểm "fake data" — chỉ trừ "demo crash". Đừng chạy live query với DB flaky.

FAQ

Rút từ FAQ workshop + dự đoán câu hỏi sau session. Click expand để đọc.

Team em dùng Streamlit / FastAPI, không dùng React. Demo có áp dụng được không?

Có. Những gì em học hôm nay là pattern, không phải framework. T-C-R (Transparency / Control / Recovery) áp dụng trên bất cứ UI nào — Streamlit, Gradio, React, HTML thuần.

Cụ thể mapping:

  • React useState → Streamlit st.session_state
  • React <details> SQL → Streamlit st.expander("SQL đã tạo")
  • React Button onClick → Streamlit st.button
  • React fetch → Streamlit gọi trực tiếp Python function

Em clone starter về, đọc logic, rồi viết lại bằng framework của nhóm. 30-60 phút là xong.

Tại sao demo hôm nay không dùng LLM thật mà dùng fake data?

Hai lý do:

  1. Demo 90 phút, không có budget cho 30 API call live — fake data = 0 cost, 0 latency surprise. Ngoài ra Gemini đôi khi rate limit khi dùng dồn trong 90 phút.
  2. Focus vào UX, không phải LLM quality — LLM tốt mà UX dở thì user vẫn bỏ. Workshop dạy UX shell, không phải fine-tuning LLM.

Thực ra demo này có call LLM thật khi VITE_LLM_API_KEY có. Fake chỉ kick in khi key missing — Tony giữ key trong .env coach, delete trong fallback scenario để demo Recovery.

Claude Code có miễn phí không? Team em đang dùng chat.claude.ai web free.

Claude Code có subscription ($20/tháng cho Pro, $200 cho Max). Nếu nhóm không mua được, alternative:

  • Cursor (có free tier) — dùng y hệt pattern
  • GitHub Copilot (student free với GitHub Education Pack)
  • Claude web + copy-paste — chậm hơn nhưng vẫn chạy được

Workshop hôm nay dùng Claude Code vì nó autonomous (tự đọc file, tự sửa, tự chạy). Các tool kia thì em phải copy-paste nhiều hơn.

Quan trọng: prompt template không đổi khi chuyển tool. Em vẫn viết "Following GUIDELINE.md section X, …" — Cursor/Copilot hiểu tương tự.

Em dùng Claude Code nhưng nó hay phá code cũ, sao giờ?

3 cách:

  1. Commit trước khi prompt — git add . && git commit -m wip → prompt → nếu phá thì git reset --hard HEAD.
  2. Prompt nhỏ, scope hẹp — thay vì "add full T-C-R", prompt từng bước (T trước, test, rồi C, test, rồi R).
  3. Đọc diff trước khi accept — Claude Code show diff, đừng bấm Accept-All. Mất 30 giây đọc, tiết kiệm 2 giờ debug.

Pro tip: trong prompt thêm "Do not touch the existing T/C UI" khi add R. Claude Code respect constraint explicit tốt hơn em tưởng.

Transparency có phải show toàn bộ prompt + reasoning cho user không?

Không. Over-transparency = user overwhelmed.

Nguyên tắc: show đủ để build trust, không phải tất cả.

  • Query-to-chart: show SQL generated, không show full chain-of-thought của LLM
  • Chat app: show source docs (3-5 refs), không show full prompt
  • Moderation: show confidence score + 1-line reasoning, không show full classifier output

Test: "user có action gì với info này?" — có action = show, không có = hide. SQL show được vì user có thể edit. Chain-of-thought show không để làm gì → hide.

Nếu em clone starter và chỉ đổi text, có bị trừ điểm vì copy không?

Không. Đây là starter officially — dùng nó là khuyến khích, không phải plagiarism. Rubric chấm agent functionality + business impact, không chấm "frontend có được code from scratch không".

Nhưng phải sửa cho phù hợp topic — không để UI mặc định "Hỏi dữ liệu bán hàng" khi app em về feedback analysis của trường học. Rubric Business Impact (25%) đánh giá "có hiểu user không" — UI generic là dấu hiệu không hiểu user.

Minimum customization: header, sample chips, system prompt, sample data. 10 chỗ tất cả.
