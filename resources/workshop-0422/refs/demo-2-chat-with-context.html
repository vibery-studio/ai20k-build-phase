Demo 2 · Chat + Context Panel

Archetype cho chat apps có cần context: trợ giảng AI, LMS chatbot có memory, gia sư Socratic, admissions / tra cứu quy định, hỏi đáp bài giảng, trợ lý nghiên cứu. Tất cả build chat app — nhưng nếu chat app không có panel bên phải, bạn đang build một phiên bản tệ hơn ChatGPT. Deep dive này gỡ ra: vì sao panel mới là defining feature, và T-C-R cụ thể trông thế nào trên shell này.

Vì sao archetype này quan trọng T-C-R breakdown
~15 topic áp dụng Baseline + 3 prompt = production-grade Same shell, nhiều domain khác nhau

TL;DR

  • Chat UI đã bị commodotise. Ai cũng copy được ChatGPT bubble trong 30 phút. Khác biệt không nằm ở message list.
  • Nhiều topic dùng chung shell. Khác domain (tutoring / admissions / Socratic / LMS chatbot / research assistant), chung archetype — chỉ thay system prompt và response bank.
  • Defining feature là panel, không phải chat. Sources + confidence bar = trust earned. Không có panel = user treat AI như black box và không tin kết quả.
Vì sao archetype này quan trọng

Chat bị commodity hoá. Panel là chỗ bạn earn trust.

Đừng nhầm: workshop này không dạy các bạn build một phiên bản ChatGPT khác. Nếu demo day các bạn show một chat bubble trả lời câu hỏi học tập — mentor đã thấy nó hai mươi lần trong tuần rồi. Cái họ chưa thấy: một chat app mà user có thể xem AI đã lấy thông tin từ đâu và AI có chắc chắn đến mức nào.

Generic chat app

Hỏi → trả lời → hết.

Bạn build một textarea + Send button + message list. Prompt Gemini. Trả về text. Render bubble. Dừng lại đó.

  • User không biết câu trả lời đến từ đâu — training data? Giả sử? Quote sai?
  • Không có cách nào biết AI đang đoán vs đang confident
  • Mỗi câu trả lời là một black box — trust giảm sau 2-3 câu sai
  • App tutoring, admissions, Socratic, LMS chatbot — tất cả nhìn y hệt nhau. Không ai nhớ được app của ai

Sai lầm không phải là code — sai lầm là design. Bạn đã build lại cái ChatGPT đã có, không kèm thứ nó thiếu.

Chat + Context Panel

Hỏi → trả lời → nguồn + confidence.

Cùng chat, nhưng split 60/40 — chat bên trái, panel bên phải. Mỗi câu trả lời của AI kéo theo 3 source card:

  • 108 (AI TA): "Sách giáo khoa Vật lý 10, Chương 2 — Động lực học chất điểm" · 94%
  • 110 (Admissions): "Quy chế tuyển sinh 2026 — Điều 12 khoản 3" · 89% (cảnh báo: deadline 15/06)
  • 114 (Socratic): Panel show reasoning path của học sinh — giáo viên theo dõi được họ đang bí đâu

Cùng một shell. Cùng src/llmService.js abstraction. Cùng T-C-R. Chỉ thay payload của panel theo domain của team — không phải rewrite UI.

Insight cốt lõi. Khi mentor demo day hỏi "app này khác ChatGPT điểm gì?" — nếu câu trả lời của bạn là "nó dạy Vật lý 10" thì mentor sẽ nói "ChatGPT cũng dạy Vật lý 10". Nhưng nếu câu trả lời là "nó show cho học sinh biết câu trả lời đến từ chương nào trong SGK, và AI confident bao nhiêu %" — đó là sản phẩm, không phải wrapper.

Panel không phải là "tính năng phụ". Panel chính là lý do archetype này tồn tại.

T-C-R breakdown

Ba prompt thêm vào baseline. Mỗi prompt một letter.

Baseline của demo 2 cố ý tối giản — chỉ chat column, không panel, không error handling, không stop button. Đó là "before" state. Coach thêm T-C-R qua 3 prompt liên tiếp, mỗi prompt một concept. Dưới đây là gì trông như thế nào, và vì sao từng chi tiết lại quan trọng.

Vì sao 3 màu? Green (>0.8), yellow (0.5–0.8), red (<0.5) là ngôn ngữ chung — user không cần học. Traffic light. User nhìn vào biết ngay "AI tin mức nào". Không có màu = user phải đọc số, cognitive load cao hơn.

Vì sao 3 nguồn max? Nhiều hơn → overwhelm. Đây không phải Google Search. 3 đủ để tạo trust, nhiều hơn là noise.

Vì sao latest-only, không phải history? Panel là context của câu hỏi hiện tại, không phải archive. Nếu user scroll lên câu cũ, panel không chạy theo — đó là intentional (simplify state, giảm tải UI).

C Control · Prompt 2

"Câu trả lời chưa chính xác?" + Dừng button.

Dưới mỗi assistant bubble, thêm link "Câu trả lời chưa chính xác?" (12px, underlined, gray). Click mở inline form: textarea "Sửa lại câu hỏi hoặc bổ sung ngữ cảnh" + button "Hỏi lại với chỉnh sửa" + "Hủy". Submit = append user message đã chỉnh, gọi lại askLLM.

Cộng thêm: "Dừng" button xuất hiện chỉ trong loading state. Click = abort request qua AbortController.

Vì sao flag button mới là key feature cho W5? Mỗi lần user click "chưa chính xác" + gõ correction = một training signal. Log cặp (original query, AI answer, user correction) vào backend — đó là golden dataset cho fine-tune hoặc retrieval tuning ở W5/W6. Không có feedback UI = bạn không có dữ liệu để improve. Chat app không có flag button = dead-end data loop.

Vì sao "Dừng" quan trọng hơn bạn nghĩ? User không nợ AI 10s attention. Nếu model chậm, họ rewrite query — "Dừng" là thoả thuận user/AI: bạn có thể thoát. Không có Dừng, user reload page → mất context cả session. AbortController là Web API native, không cần library.

R Recovery · Prompt 3

Retry button + short-query warning.

Wrap askLLM trong try/catch. Error → append assistant message với flag error: true. Render bubble đỏ với ⚠ emoji, error text tiếng Việt, và button "Thử lại" re-run câu query gần nhất.

Input validation: query trim < 5 ký tự → KHÔNG gọi LLM. Hiện warning cam dưới input: "Câu hỏi quá ngắn — hãy viết rõ hơn (tối thiểu 5 ký tự)." Clear khi user gõ thêm.

Panel giữ source card của câu trả lời thành công gần nhất. Không clear khi có error — user vẫn có context cũ để đối chiếu.

Vì sao <5 ký tự không trigger LLM? Ba lý do: (1) Tiết kiệm tiền — mỗi call Gemini là token cost, call "aa" là phí đốt. (2) Dạy user cách prompt — warning cam mềm là coaching, không phải block. (3) Clean data loop — log chỉ có meaningful query, analytics về "câu hỏi phổ biến" không bị nhiễu bởi test junk.

Vì sao error bubble màu đỏ giữa chat thread? Visible nhưng không dừng UX. Modal error = user phải dismiss trước khi làm gì tiếp → friction. Bubble đỏ inline = user đọc, click "Thử lại", tiếp tục. Đặc biệt quan trọng cho admissions / advising apps — khi user đang gõ câu hỏi về deadline, error modal làm họ mất đà.

Stack decisions

Cùng abstraction llmService.js. Contract khác.

Cả 3 demo đều share pattern "UI chỉ import askLLM, không call SDK trực tiếp". Nhưng contract của Demo 2 khác Demo 1 và Demo 3 — vì output không phải SQL + rows, mà là answer + structured sources.

Contract askLLM signature 
// src/llmService.js

export async function askLLM(
  query: string,
  history: Array<{
    role: "user" | "assistant",
    content: string
  }>
) : Promise<LLMResponse>

// LLMResponse:
{
  answer: string,
  sources: [
    {
      title: string,
      snippet: string,
      confidence: number  // 0..1
    },
    // max 3 items
  ]
}

PHẢI trả về JSON strict:
{
  "answer": "...",
  "sources": [
    {"title": "...",
     "snippet": "...",
     "confidence": 0.85}
  ]
}

// defensive parse:
try { return JSON.parse(rawText) }
catch { return {
  answer: rawText,
  sources: []
}}

Vì sao JSON strict + fallback? Gemini 2.5 Flash đôi khi wrap JSON trong markdown code fence (```json...) hoặc thêm commentary. Nếu bạn call JSON.parse thẳng → crash. Giải pháp: luôn bọc trong try/catch, nếu parse fail thì fallback sang {answer: rawText, sources: []}. UI không crash, user vẫn thấy text — chỉ mất panel context (graceful degradation).

Vì sao env vars tên chung chung VITE_LLM_* thay vì VITE_GEMINI_*? Vì bạn sẽ swap provider. W4 dùng Gemini (free tier). W5 Tony có thể đổi sang Claude Haiku hoặc Llama local. Nếu đặt tên env theo provider → phải rename mỗi lần đổi → dễ miss một chỗ → bug. Tên provider-neutral = sửa một file llmService.js, xong.

Adapt cho team của bạn

Cùng shell. Ba domain. Thay payload, không thay code.

Đây là phần các bạn nên đọc kỹ nhất. Archetype không phải là "copy demo rồi đổi title". Archetype là hiểu được: source, confidence, và panel state mỗi team có nghĩa gì khác nhau — nhưng cấu trúc code không đổi.

Anti-patterns

5 lỗi hay gặp nhất khi build archetype này.

Coach đã thấy 5 lỗi này trong 20+ cohort trước. Mỗi lỗi đều có vẻ đúng lúc design, nhưng kill trust sau 2-3 câu hỏi user.

×Show full chain-of-thought

"User càng biết nhiều càng trust" — sai. Show toàn bộ reasoning của AI (tất cả step, mọi cân nhắc) làm user overwhelmed. Họ cần action-able info, không phải thesis.

Fix: show đủ để build trust, không phải tất cả. Chat app: 3 source card. Không phải 15 ref + full chain-of-thought + self-critique.

×Hide sources entirely

"Chat clean hơn khi không có panel" — clean với ai? User không biết AI đọc ở đâu → treat mọi câu trả lời như truyền miệng. Trust về 0 sau câu sai đầu tiên.

Fix: panel có thể collapse, nhưng phải có. Default state = expanded. Collapse = progressive disclosure, không phải hide completely.

×Fake confidence numbers

"Hard-code 95% cho tất cả" để UI nhìn xịn. User click 3 câu thấy đều 95% → nhận ra số giả → mất niềm tin hoàn toàn. Giả confidence tệ hơn không có confidence.

Fix: dùng self-rated confidence từ Gemini system prompt. Hoặc skip confidence bar hẳn. KHÔNG BAO GIỜ hard-code 95% giả. Người dùng smarter than that.

×Long loading spinner, no status text

Spinner xoay 10 giây không kèm chữ = user assume hỏng. Đặc biệt tệ cho demo day — mentor sẽ click X trước khi response về.

Fix: placeholder bubble với "AI đang trả lời..." (italic, grayed). Thêm "Dừng" button xuất hiện sau 2s loading. User biết đang live, có option thoát.

×Error crashes UI

Gemini 429 rate limit → JSON.parse fail → React error boundary → white screen. Demo day coach chứng kiến nhiều nhất. Recovery prompt (prompt 3) fix đúng lỗi này.

Fix: wrap askLLM trong try/catch. Error = bubble đỏ inline với "Thử lại" button. Không bao giờ để lỗi propagate lên UI level. Graceful degradation, không crash.

×Panel chạy theo mọi message cũ

Build panel "smart" — click message nào trong history, panel update sources của message đó. Nghe hay, thực tế làm state explode, dễ bug, user confused vì panel đổi mỗi khi họ scroll.

Fix: panel chỉ show sources của câu trả lời mới nhất. Đơn giản, state ít, user không bao giờ lạc. Smart UX không phải nhiều tương tác — là ít cognitive load.
Production upgrade path

W4 MVP → W5 real RAG → W6 streaming + edit history.

Baseline + T-C-R là đủ cho W4 demo. Nhưng nếu bạn muốn ship production-grade cho demo day (hoặc pivot thành sản phẩm thật), đây là roadmap 3 tuần từ đây.

Tuần 5 · Real RAG

Backend + retrieval thật

Thay hard-coded sources bằng retrieval thật từ KB của team.

  • Upload SGK/policy PDF lên vector DB (Pinecone / Qdrant / local Chroma)
  • Backend endpoint POST /ask: embed query → top-k retrieve → pass context vào Gemini system prompt
  • Gemini trả JSON với real sources từ KB (không bịa)
  • Confidence = cosine similarity của top-k, không phải self-rated
  • Flag button ghi log vào feedback_logs table cho fine-tune sau

UI không đổi. Chỉ thay llmService.js — trỏ về backend của bạn.

Tuần 6 · Polish

Streaming + edit history

Làm chat feel natural, thêm audit trail.

  • Token streaming — Gemini generateContentStream, render incrementally
  • Per-message edit history: user có thể scroll lại message cũ, xem phiên bản trước khi edit
  • Citation highlighting — click source card, scroll chat xuống đoạn citations
  • Mobile responsive (panel collapse thành bottom sheet)
  • Markdown render cho code / formula (nếu topic dạy Toán / coding)

Tùy chọn. Demo day cần nhất là W5 hoạt động — W6 là bonus.

Quy tắc: không skip W5 để nhảy thẳng W6. Streaming + polish trên fake data là AI-slop — đẹp nhưng rỗng. Demo day, mentor sẽ click source card và hỏi "thật không?". Nếu sources hard-coded → lộ ngay. Priority: W5 > W6.

Khi nào skip streaming? Nếu response trung bình < 3s, streaming không đáng đầu tư complexity. Loading bubble 2s đủ OK. Streaming có lợi khi response > 5s — user thấy progress.

FAQ

Câu hỏi thực tế từ cohort trước.

Lọc từ faq.md — các câu hỏi riêng cho archetype này. Category E (T-C-R context) là thứ mentor hỏi nhiều nhất.

E1 App em là Socratic Tutor (114), em cố ý không trả lời — T-C-R áp dụng sao?

Perfect case thực ra. T-C-R càng quan trọng khi agent "từ chối":

  • Transparency: show cho user thấy AI đang dẫn dắt thay vì trả lời — ví dụ badge "Socratic mode — tôi sẽ hỏi lại thay vì trả lời"
  • Control: cho user bật/tắt Socratic mode (ít nhất cho debug) — "nếu bạn thực sự cần đáp án, gõ /answer"
  • Recovery: khi user frustrated, có "tôi đang bí" → AI chuyển sang mode hint

Không có 3 cái này → user nghĩ AI hỏng.

E3 Transparency có phải show toàn bộ prompt + reasoning cho user không?

Không. Over-transparency = user overwhelmed.

Nguyên tắc: show đủ để build trust, không phải tất cả.

  • Chat app: show source docs (3-5 refs), không show full prompt
  • Text-to-SQL: show SQL generated, không show full chain-of-thought
  • Moderation: show confidence score + 1-line reasoning, không show full classifier output

Test: "user có action gì với info này?" — có action = show, không có = hide.

E4 Có cần giải thích T-C-R cho user không, hay tự user hiểu?

Tự user hiểu. T-C-R là cho em design, không phải label trong UI.

User không bao giờ thấy chữ "Transparency" — họ chỉ thấy "à, tôi biết AI đang nghĩ gì, tôi tin tưởng". Đó là T thành công.

C5 Tại sao demo hôm nay dùng Gemini thật (có API key) mà không dùng hoàn toàn fake data?
