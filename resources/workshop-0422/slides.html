UI pattern · 01 / 07
Pattern 01

Chat + context panel

Trợ giảng AI & Q&A bài giảng Gia sư Socratic Tra cứu quy định & admissions LMS chatbot có memory

User hỏi, AI trả lời ở cột trái. Cột phải có panel — để AI show nguồn tham khảo, tiến độ học, hoặc cách nó suy luận.

Câu hỏi UX quan trọng Panel bên phải nên show cái gì? Trợ giảng AI thì hiện tiến độ bài học của sinh viên. Admissions thì hiện link quy định được trích. Gia sư Socratic thì hiện badge "chế độ Socratic". Chọn một thứ user thực sự cần — đừng dồn hết vào panel.

UI pattern · 02 / 07 · Deep-dive ref
Pattern 02

Upload → dashboard

Phân tích feedback SV OCR bài viết tay Ảnh kính hiển vi / Medical VLM Rà soát CAD / bản vẽ Số hóa bảng trắng

User upload file lên — CSV feedback, audio ghi âm, PDF. AI xử lý cả đống, rồi trả về một dashboard: thống kê tổng, vài trích dẫn nổi bật, biểu đồ xu hướng.

Câu hỏi UX quan trọng Có cho user xem dữ liệu gốc không? Giấu hết — user không tin kết quả AI đưa ra. Show hết — user ngợp. Cân bằng: hiện 3–5 ví dụ + thống kê tổng, ai muốn xem sâu thì click vào.

UI pattern · 03 / 07 · Deep-dive ref
Pattern 03 · Interactive walkthrough

Query → structured result

Text-to-SQL / NL → data Voice → dashboard Digital twin đại học Search engine / semantic Knowledge graph query

User hỏi bằng ngôn ngữ tự nhiên, AI trả về dữ liệu có cấu trúc — bảng, biểu đồ, hoặc kết quả SQL. Bữa nay không code live, nhưng có bản interactive ở link dưới.

Câu hỏi UX quan trọng Có hiện câu SQL (hoặc logic) mà AI viết ra cho user xem không? Hiện — user tin hơn, nhưng cần biết đọc SQL. Không hiện — trông như magic, sai thì user không biết sửa. Lựa chọn này phụ thuộc user của bạn: admin hay người dùng cuối.

UI pattern · 04 / 07 · Deep-dive ref
Pattern 04

Wizard + inline audit

Soạn syllabus / ABET Soạn giáo án GDPT 2018 Tạo đề thi Bloom Viết nhận xét học bạ Thiết kế CTĐT

User điền form theo từng bước. AI đọc ngay trong lúc điền, thấy sai chuẩn là báo luôn — không phải chờ bấm Submit mới biết.

Câu hỏi UX quan trọng Khi AI phát hiện lỗi — chặn user đi tiếp, hay chỉ tô đỏ cảnh báo? Chặn nếu là chuẩn bắt buộc (ABET, pháp lý). Chỉ cảnh báo nếu là gợi ý (giáo viên có thể tự quyết). Nhầm hai cái này — user sẽ bực và bỏ app.

UI pattern · 05 / 07 · Deep-dive ref
Pattern 05

Draft → approve → send

Truyền thông khẩn cấp Giao tiếp phụ huynh Xử lý đơn từ tự động Onboarding nhân viên Social commerce agent

AI viết sẵn tin nhắn. User đọc, sửa, duyệt, rồi bấm Gửi. Tin đi qua Zalo / SMS / Email — gửi xong không thu lại được. Ngoài đời không có Ctrl-Z.

Câu hỏi UX quan trọng Bước xác nhận nên thiết kế thế nào? Bấm 2 lần? Gõ chữ "CONFIRM"? Xem preview từng kênh? Lưu ý: SMS, Email, Push notification hiện ra khác nhau trên điện thoại — mỗi kênh nên có một preview riêng để user quét nhanh trước khi gửi.

UI pattern · 06 / 07 · Deep-dive ref
Pattern 06 · Interactive walkthrough

Queue + approval dashboard

Moderation cộng đồng Chấm bài quy mô lớn Cảnh báo sớm SV Rà soát hợp đồng / chính sách Liêm chính học thuật

AI xử lý hàng loạt trước — phân loại, chấm điểm, gắn flag. Người kiểm duyệt mở dashboard, duyệt từng cái theo queue. Ở đây chưa code live, link dưới có bản interactive.

Câu hỏi UX quan trọng Thiết kế cho reviewer làm nhanh. Cần 3 thứ: filter để lọc, action hàng loạt để khỏi click từng cái, và sắp xếp theo confidence — cái AI phân vân nhất lên đầu (cần người xem), cái chắc chắn đẩy xuống dưới. Phím tắt bàn phím = nâng tốc độ 3–5 lần.

Pattern 07

Real-time streaming

Luyện phỏng vấn / avatar AI Huấn luyện thể thao qua video Q&A hội thảo voice Thí nghiệm ảo multi-agent Hệ thống luyện tiếng Anh

App xử lý thời gian thực — voice, audio live, hoặc AI trả lời từng chữ một. Mọi thứ đều phải nhanh, độ trễ thấp.

Câu hỏi UX quan trọng Trong lúc AI đang xử lý, user thấy gì? Transcript đang gõ từng chữ? Thanh progress? Hoạt ảnh "đang suy nghĩ"? Chọn gì cũng được, chỉ cần user không bao giờ nhìn màn hình trống. Trống = user tưởng app hỏng.

T · Transparency Bước 2

Mỗi câu trả lời có chủ đề + mức tin cậy.

Panel bên phải hiện 1-2 chủ đề (topic tag từ danh sách cố định) và một confidence bar gắn label "AI tự đánh giá". Đây không phải trích dẫn nguồn — đó là classification. Gọi đúng tên để user không hiểu nhầm là có sách thật đứng sau.

C · Control Bước 3

AI trả lời sai? User có đường sửa.

Dưới mỗi câu có link "Câu trả lời chưa chính xác?" — user ghi lý do rồi submit, flag đó hiện trên teacher list. Đang loading thì có nút Stop để hủy request.

[C] Add Control: "Câu trả lời chưa chính xác?" link below each AI message opens inline form (reason textarea + submit), "Stop" button during loading uses AbortController to cancel fetch. Flagged answers get small "đã báo cáo" badge.

R · Recovery Bước 4

Chặn lỗi trước khi nó xảy ra.

User gõ câu quá ngắn hoặc mơ hồ — cảnh báo nhưng không block, user vẫn gửi được. Call lỗi thì render bubble đỏ có nút Thử lại.

[R] Add Recovery: try/catch around llmService call, error renders red bubble with "Thử lại" button that resubmits last user message. Pre-flight: if input < 5 chars show orange inline warning "Câu hỏi hơi ngắn, thêm ngữ cảnh giúp AI trả lời tốt hơn" — warn only, do NOT block submit.
